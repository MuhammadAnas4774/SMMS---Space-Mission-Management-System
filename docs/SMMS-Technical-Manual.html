1. Executive overview

The Space Mission Management System (SMMS) is a browser-based command console for orchestrating personnel, missions, spacecraft, orbital infrastructure, experiments, telemetry, and alerts. It implements a PRD §7.x route catalog with both specialized screens (dashboard, persons + subtype flow, multi-tab creators) and a generic CRUD shell for junction and auxiliary tables.

Core capabilities

  • Mission Control Dashboard — live UTC clock, KPI stat chips, Chart.js mission status bar chart, entity bar chart, synthetic telemetry sparkline, recent alerts grid.
  • Person lifecycle — optional Astronaut / Ground Control subtype creation with validation and delete-impact preview endpoint.
  • Typed spacecraft & modules — Capsule / Shuttle / Rocket and Habitat / Laboratory flows per table-per-type (TPT) patterns.
  • Experiment taxonomy — Biological, Physical, Astronomy, Engineering experiment subtype records.
  • Unified REST layer — parameterized SQL via entity registry (backend/src/entities.js) drives list/filter/sort/create/update/delete.

2. System architecture

The solution follows a classical three-tier pattern:

  1. Presentation — React 19 functional components, client-side routing, Axios HTTP, contextual toast notifications.
  2. Application — Express middleware pipeline with JSON body parsing, centralized entity routing, and specialized handlers registered before generic parameterized routes.
  3. Persistence — InnoDB tables with declarative FK constraints, cascades for subtype ownership, and restrictive deletes where junction rows reference parents.

4. Database design

Schema script backend/sql/schema.sql provisions database smms and applies table-per-type extensions for spacecraft (Capsule, Shuttle, Rocket), modules (Habitat, Laboratory), and experiments (Biological, Physical, Astronomy, Engineering_exp). Telemetry splits into parent TELEMETRY_DATA plus subtype payloads (SYSTEM_STATUS, ENVIRONMENTAL_DATA, NAVIGATION_DATA).

4.1 Primary entities

PERSONHuman identity root; referenced by ASTRONAUT and GROUND_CONTROL (1:1).

4.2 Referential integrity highlights

  • Astronaut subtype references PERSON with ON DELETE CASCADE — removing a person clears subtype projections.
  • Generic DELETE endpoint surfaces MySQL errno 1451 as HTTP 409 when dependents block removal.
  • Optional mentor linkage inside ASTRONAUT self-references with ON DELETE SET NULL.

5. REST API & CRUD engine

Dynamic routes resolve :entity keys against entities.js. Identifier quoting uses backticks to safely compose SQL identifiers while values remain bound parameters (injection-safe pattern).

5.1 Generic CRUD

MethodPathBehaviour
GET/api/:entityPaginated list (page, limit), optional search (LIKE across whitelist), column filters, sort (sort, dir).
GET/api/:entity/:idSingle row by primary key.
POST/api/:entityInsert JSON body; returns hydrated row (SELECT after insert).
PUT/api/:entity/:idPartial update excluding PK column.
DELETE/api/:entity/:idHard delete; FK violations mapped to friendly conflict message.

7. Feature deep-dive

7.1 Dashboard analytics

  • MissionStatusChart — vertical bar chart of mission counts by status (grouped SQL); avoids doughnut canvas sizing issues in layout.
  • EntityBarChart — top-N table cardinality snapshot.
  • TelemetryLineChart — illustrative dual-series timeline blending mission & alert signals.

7.2 Theming & UX polish

  • CSS variables in index.css establish glassmorphism, neon accents (#6366f1 / #22d3ee), and motion primitives.
  • Lucide icons unify sidebar + dashboard chips (replacing legacy emoji accents).
  • SystemOverviewCard — centered overview narrative, live UTC mission clock, and creator attribution (no heavy decorative graphics).

7.3 Operational safeguards

  • BigInt JSON serialization shim on Express json replacer.
  • SQL identifier hardening via quoteIdent.
  • Startup connectivity banner guiding schema import when DB absent.

8. Installation & operations

Database bootstrap

  1. Execute backend/sql/schema.sql then backend/sql/seed.sql (phpMyAdmin import, Workbench, or CLI redirection).
  2. Copy backend/.env.example → .env; align credentials with MySQL.

Development servers

cd "Database Project"
npm install
npm run dev          # concurrently API :5000 + Vite :5173

Production build

npm run build:frontend   # outputs frontend/dist static assets

Serve dist/ behind any static host; configure reverse proxy to backend API origin.
